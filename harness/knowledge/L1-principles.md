# L1 — 开发价值观（跨阶段通用原则）

> **L1 是跨阶段恒为真的开发价值观**：在 clarify / plan / build / verify / retrospect 任意阶段都成立，
> 回答「怎么做事」，不回答「这一步做什么」（后者是 [`L2-stage-rules.md`](L2-stage-rules.md)）。
>
> 这些原则从跨项目开发经验中抽象而来，**语言无关**——已剔除一切框架/技术栈专有名词。
> 每条带稳定编号 `P{n}`，作为 L2 `derives` 引用与 lessons 晋升的锚点。
> 三者关系与晋升通道见 [`README.md`](README.md)。

---

## P1 ｜tags: change-discipline, reversible
- **原则**：小步快走——变更增量式推进，优先可回滚、可验证。
- **为何**：小步子让每一步都能独立验证，出错时回退代价小，不会一次推翻一大片。
- **落到执行**：plan 阶段把需求拆成能独立交付的 Phase；build 阶段一个 Phase 一个 commit。
- **来源**：workbook（跨项目通用沉淀）

## P2 ｜tags: scope, caution
- **原则**：按需操作——仅在必要时读取或修改文件，避免大范围无确认的改动。
- **为何**：读改面越克制，越不容易引入意外副作用，也越省上下文。
- **落到执行**：动手前先确认要碰哪些文件；不顺手扩展、不夹带计划外改动。
- **来源**：workbook（跨项目通用沉淀）

## P3 ｜tags: change-discipline
- **原则**：最小改动——每次修改只涉及必要的代码，避免过度重构或大规模重组。
- **为何**：变更面越小，可验证性和可回滚性越高，评审也越容易看清。
- **落到执行**：build 写码前先圈定「必改集合」；review / scope-guard 据此判定是否越界。
- **来源**：workbook（跨项目通用沉淀）

## P4 ｜tags: change-discipline, stability
- **原则**：新增优于修改——优先通过新增文件/函数/类实现新功能，而非改动现有代码。
- **为何**：新增不破坏既有稳定路径；改动现有代码则可能波及所有调用方。
- **落到执行**：能加一个新函数解决的，就别去改老函数的签名或内部逻辑。
- **来源**：workbook（跨项目通用沉淀）

## P5 ｜tags: traceability
- **原则**：可追溯——任何自动化操作都应留下记录，执行后回报结果与记录位置。
- **为何**：可追溯的过程才能被复查、复用、追责，也便于交接。
- **落到执行**：与 min-spec 的 JSONL 留痕（`scripts/report_hook.py`）同源——阶段切换、门禁结论都落痕。
- **来源**：workbook（跨项目通用沉淀）

## P6 ｜tags: compatibility, interface
- **原则**：向后兼容——新功能不应破坏现有接口；若需接口变更，须先讨论并更新所有调用方。
- **为何**：悄悄改接口会让调用方在远处崩坏，且难以定位。
- **落到执行**：接口变更要在 clarify/plan 阶段明确列入影响面，且改全所有调用点。
- **来源**：workbook（跨项目通用沉淀）

## P7 ｜tags: structure, single-responsibility
- **原则**：单一职责——一个文件一个功能，看名字能推测出它做什么。
- **为何**：职责单一的文件易理解、易测试、易复用，改动影响范围清晰。
- **落到执行**：不要把不相关的功能混进一个文件；新功能不确定归属时先想清楚再落文件。
- **来源**：workbook（跨项目通用沉淀）

## P8 ｜tags: read-before-write, understanding
- **原则**：前置代码阅读——改动或扩展某模块前，必须先通读相关现有实现。
- **为何**：不理解现有接口约定、依赖链、错误处理与设计模式就动手，极易重复造轮子或违反既有设计。
- **落到执行**：与 review SKILL「先独立建立基线、再看代码」同源——读懂了再写。
- **来源**：workbook（跨项目通用沉淀）

## P9 ｜tags: communication
- **原则**：沟通——需要用户选择/确认时，先给出清晰选项与各自优缺点，再一次只问一个问题，持续追问到信息足够明确。
- **为何**：一次抛一长串问题会淹没用户；不给优缺点会导致反复沟通。
- **落到执行**：与 clarify SKILL「一轮最多问 3 个、先问最致命」同源；不清楚就问，绝不假设。
- **来源**：workbook（跨项目通用沉淀）

## P10 ｜tags: single-source-of-truth, docs
- **原则**：唯一事实源——同一事项只保留一个权威文档，禁止并行记录同类状态，中间成果须回填。
- **为何**：双份事实源必然分叉，交接时不知该信哪个。
- **落到执行**：与 min-spec「docs/meta 各类产物各有唯一落点」同源；临时辅助文件用完合并回主文档。
- **来源**：workbook（跨项目通用沉淀）
