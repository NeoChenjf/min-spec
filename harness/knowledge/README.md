# knowledge — 分层知识库

> min-spec 的**跨项目通用知识库**。随 harness 模板分发（`cp -r harness` 自动带上），
> 是给每个阶段子 Agent 喂的「开发价值观 + 阶段操作规则」。

## 三个文件

| 文件 | 是什么 | 取值 |
|---|---|---|
| [`L1-principles.md`](L1-principles.md) | 跨阶段恒为真的开发价值观（怎么做事） | `P1`~`Pn` |
| [`L2-stage-rules.md`](L2-stage-rules.md) | 绑定到具体阶段的可执行 checklist（这一步做什么） | `S1`~`Sn`，每条带 `适用阶段` |
| 本 README | 三者关系 + 晋升通道 | — |

L1/L2 的分层判据是**「是否绑定阶段」**这个客观维度：
- 在任意阶段都成立、与具体阶段动作无关 → L1。
- 能说出「在某阶段开工/收尾时执行」 → L2（并标注 `适用阶段`，必须是 DAG 阶段名）。

L2 用 `derives` 引用它落地的 L1 编号，**不复述** L1 内容（守唯一事实源 P10）。

---

## 与 `docs/meta/knowledge/lessons.md` 的关系（关键）

min-spec 里有两处「知识」，性质不同，**严格单向数据流**，不可混淆：

| | L1/L2（`harness/knowledge/`） | `lessons.md`（`docs/meta/knowledge/`） |
|---|---|---|
| 作用域 | 跨项目通用 | 单项目本地 |
| 稳定性 | 稳定、版本化 | 持续增长 |
| 是否随模板分发 | **是**（在 harness/ 下） | **否**（运行时产物） |
| 写入者 | 人工 + 晋升流程（谨慎） | retrospect 阶段自动追加（高频） |

**一句话**：`lessons.md` 是 L1/L2 的**候选池**，L1/L2 是 lessons 的**毕业生名册**。
retrospect 永远只写 `lessons.md`，**绝不直接写本目录**——晋升是唯一的跨界写入，且须人工把关。

## 晋升通道：lessons → L1/L2

retrospect 沉淀的某条 lesson，若足够通用，可「晋升」成 L1/L2 的正式条目。

### 晋升三准则（全部满足才晋升）
1. **跨项目成立**：换一个语言/项目仍然有效——去掉所有框架/技术栈专有名词后还剩下站得住的内容。
2. **复现 ≥ 2 次 或 后果严重**：在 `lessons.md` 出现过等价条目两次以上，或单次但属高代价踩坑。
3. **不与现有 L1/L2 重复**：已有等价 `P`/`S` 条目则不晋升。

### 晋升动作（人工触发、可审计）
1. **retrospect 阶段只「标记候选」**：在 `{run_id}-retro.md` 里列出晋升候选（建议归 L1 还是 L2、若 L2 标注阶段），**不在该阶段执行写入本目录**——晋升是跨项目动作，须人工确认。
2. **人工确认后写入**：把候选改写成 `P{n}` / `S{n}` 格式，追加进对应文件，并在该文件「变更日志」记一行：`{date} 晋升：L-{run_id}-{序号} → S{m}`。
3. **反向标记**：被晋升的 lesson 在 `lessons.md` **保留原文不删**（它是历史证据，唯一事实源），并标注「已被 P{n}/S{n} 覆盖」；retrospect 去重时发现已被覆盖即不再重复入库。

这样：高频自动写入只发生在 `lessons.md`（无审查成本）；本目录的写入永远经人工三准则把关（保证通用与稳定）。两条写路径永不交叉。

## 校验

`scripts/validate.py` 对本目录只做**结构审计**，不解析规则正文：
- 三个文件必须存在（在 `REQUIRED_FILES` 里）。
- `L2-stage-rules.md` 每条 `适用阶段` 标签必须 ∈ DAG 阶段名（`check_knowledge`）。

条目数量、tags、derives 指向是否存在、lessons↔L1/L2 是否同步——**不入门禁**，交人工三准则治理。
